## 第2页 选题背景与研究意义

### 页面要点

- 日常学习和项目记录产生大量碎片化笔记，后续整理成本高
- 传统笔记系统多以目录管理为主，难以表达知识之间的关联关系
- 当笔记规模扩大后，仅靠标题或目录查找效率较低
- 需要构建“记录、关联、检索、可视化”一体化的个人知识管理工具

### 建议讲稿

本课题来源于真实的学习和知识整理需求。随着课程学习、技术积累和项目实践不断推进，个人会形成大量笔记，但这些内容往往分散在不同目录中，缺乏有效联系。传统笔记工具更擅长存储，却不擅长建立知识网络。因此，我希望通过双链机制、检索能力和图谱展示，把分散知识点连接起来，提高后续查找、复习和复用效率。

## 第3页 系统目标与功能设计

### 页面要点

- 用户认证与基础设置：注册、登录、资料维护、主题配置
- 知识库与目录管理：知识库、文件夹、笔记树形组织
- Markdown 编辑与自动保存：支持实时编辑和内容持久化
- 双链与历史版本：支持 `[[笔记标题]]` 引用和版本追踪
- 检索与模板复用：支持关键词、标题、正文、标签检索及模板套用
- 知识图谱可视化：直观展示笔记之间的关联结构

### 建议讲稿

围绕个人知识管理闭环，我将系统拆分为六类核心功能。第一类是用户与基础设置，解决登录和个性化配置问题；第二类是知识库与目录管理，用于组织内容；第三类是 Markdown 编辑与自动保存，保证日常记录体验；第四类是双链和历史版本，实现关联与追踪；第五类是检索与模板，提高内容复用和定位效率；第六类是知识图谱，用于从整体上观察知识结构。

### 页面要点

- 核心表包括：用户表、用户设置表、知识库表、文件夹表、笔记表、正文表、双链表、历史版本表、模板表、标签表、标签关联表

- 知识库、文件夹、笔记均设计软删除字段，为后续回收站预留基础

### 建议讲稿

数据库围绕“用户、知识库、文件夹、笔记、双链关系”进行设计。这里有三个关键点。第一，笔记标题和正文分表存储，这样在自动保存时可以分别更新标题和内容；第二，双链关系独立建表，既方便做入链和出链统计，也为图谱节点边数据生成提供基础；第三，系统预留了软删除字段，为后续实现回收站和恢复机制提供可扩展空间。

## 第6页 关键实现：Markdown 编辑、自动保存与双链解析

### 页面要点

- 使用 Vditor 作为 Markdown 编辑器，支持富文本、代码块、公式和导出
- 标题自动保存接口与正文自动保存接口分离，减少冗余更新
- 正文保存时解析 `[[笔记标题]]` 结构，自动维护双链关系
- 若目标笔记不存在，则标记为失效双链，后续可在笔记补全后自动恢复
- 笔记详情页可返回入链、出链、候选项和链接预览信息

### 建议讲稿

这一页是项目的核心。用户在编辑器中输入 Markdown 内容后，系统会通过自动保存接口将正文写入数据库。保存过程中，后端会使用正则表达式解析 `[[笔记标题]]` 形式的引用，并刷新当前笔记对应的双链关系。如果目标笔记已经存在，则建立正常链接；如果目标笔记尚不存在，则先标记为失效双链。这样既保证编辑体验，又让双链关系和图谱数据始终与正文保持同步。

## 高频答辩问题参考

### 1. 为什么要做双链，而不是只做标签或者目录？

目录更适合层级组织，标签更适合分类检索，但二者都不擅长表达“知识点之间的引用关系”。双链的价值在于把原本分散的笔记连接成知识网络，让用户不仅能存内容，还能追踪内容之间的上下文关系，因此更适合构建个人知识体系。

### 2. 你的双链是怎么实现的？

我在后端正文保存逻辑中解析 `[[笔记标题]]` 结构。保存时先删除当前笔记旧的出链，再重新解析正文并生成新的链接记录。如果目标笔记存在，则记录正常链接；如果不存在，则标记为失效链接。这样能保证双链数据始终与正文内容保持一致。

### 3. 为什么笔记标题和正文要分表？

因为标题和正文的更新频率、长度和业务作用不同。标题更适合用于目录和快速定位，正文可能较长且更新更频繁。分表后更方便实现标题自动保存、正文自动保存和后续扩展，也有利于保持数据结构清晰。

### 4. 知识图谱的数据来源是什么？

图谱节点来源于知识库下的笔记数据，图谱边来源于双链关系表。后端会统计每个节点的入链和出链数量，再把节点和边统一封装成图谱数据接口返回前端，前端使用 D3 进行可视化布局和交互展示。

### 5. 项目当前最大的不足是什么？

我认为最大的不足主要有两个。第一是工程化能力还可以继续提升，比如自动化测试、备份同步和回收站闭环仍未完全完成；第二是知识图谱目前更多是“展示型”能力，后续还可以继续向局部关系分析、路径探索和更强的交互能力扩展。
