- 用户与基础设置模块主要包括用户登录与注册、用户基本信息管理以及系统个性化配置等功能。用户通过登录系统即可使用笔记编辑、双链关联与知识可视化等核心功能，新用户可通过注册建立个人知识空间。登录后的用户能够对个人基本信息进行维护，并根据自身使用习惯对界面显示、编辑器行为等内容进行配置，从而提升系统的个性化使用体验。（有时间未来可能会支持用户自定义CSS样式。） 
- Markdown 编辑与渲染模块是系统的重要基础功能模块，主要用于支持笔记内容的高效编写与实时展示。该模块支持 Markdown 语法的实时解析与渲染，并对数学公式、代码块、图片和视频等内容提供良好支持。同时，系统通过快捷键输入增强编辑效率，并针对双链内容进行渲染，使笔记在编辑状态与阅读状态下均保持良好的一致性与可读性。 
- 知识库、文件夹、笔记管理模块已经形成以知识库为归属边界、文件夹为层级组织、笔记为核心内容单元的基础管理闭环。后端支持知识库与文件夹的创建、查询、更新、删除，并能够将多级文件夹与根目录笔记组装为统一目录树；笔记侧已支持新建、详情查询、编辑、移动、标题与正文自动保存、删除，以及基于模板初始化内容。与此同时，系统还提供历史版本的手动创建、查看、恢复与删除能力，并通过软删除字段维护知识库、文件夹、笔记的状态，从而兼顾日常整理效率与后续扩展恢复机制。 
- 双链与知识关联模块用于实现笔记之间的关联关系构建与维护。用户在编写笔记时，可以通过内部链接方式将不同笔记进行引用，系统会自动生成双向链接关系，并支持反向链接的展示与更新。还具备链接自动补全与提示功能，帮助用户快速建立笔记间的关联结构，从而逐步形成互相连接的知识网络。 
- 知识结构与可视化模块主要用于对笔记及其关联关系进行可视化展示。系统以知识图谱的形式呈现全部或局部笔记节点，直观反映当前笔记之间的关系结构。用户可以查看当前笔记的局部关联图，也可以浏览整体知识网络，并通过图形界面实现节点跳转与知识路径探索，加深对知识体系整体结构的理解。
- 检索与导航模块为用户提供高效的信息定位与内容查找能力。该模块支持全文内容搜索、标题搜索以及标签搜索等多种方式，并可结合筛选条件进行高级检索。通过双链关系的整合，系统还能够辅助用户在不同笔记之间快速导航，提高在大规模笔记库中的查找效率。 
- 导出与同步模块主要负责笔记数据的导出、备份与同步功能。系统支持将笔记内容导出为 PDF 等格式，并通过自动保存机制保障数据安全。实现数据同步，提升系统的可靠性与可持续使用能力。

#### 1. 用户注册

##### 接口说明

- 接口路径：POST /register
- 请求方式：POST
- Content-Type: application/json
- 是否需要登录：否

##### 请求参数

| 参数名          | 类型   | 必填 | 说明     | 校验规则       |
| --------------- | ------ | ---- | -------- | -------------- |
| username        | String | 是   | 用户名   | 3-20 个字符    |
| password        | String | 是   | 密码     | 6-20 个字符    |
| confirmPassword | String | 是   | 确认密码 | 必须与密码一致 |
| email           | String | 是   | 邮箱     | 有效邮箱格式   |
| nickname        | String | 是   | 昵称     | 最多 20 个字符 |

##### 请求示例

```json
{
  "username": "testuser",
  "password": "123456",
  "confirmPassword": "123456",
  "email": "test@example.com",
  "nickname": "测试用户"
}
```

##### 响应参数

| 参数名           | 类型    | 说明                |
| ---------------- | ------- | ------------------- |
| code             | Integer | 状态码              |
| message          | String  | 响应消息            |
| data             | Object  | 返回数据            |
| data.id          | Long    | 用户 ID             |
| data.username    | String  | 用户名              |
| data.email       | String  | 邮箱                |
| data.nickname    | String  | 昵称                |
| data.avatarUrl   | String  | 头像 URL            |
| data.status      | Integer | 状态：1 正常 0 禁用 |
| data.createdTime | String  | 创建时间            |
| data.updatedTime | String  | 更新时间            |
| data.token       | String  | JWT 令牌            |

##### 响应示例

```json
成功响应：
{
  "code": 200,
  "message": "注册成功",
  "data": {
    "id": 123456789,
    "username": "testuser",
    "email": "test@example.com",
    "nickname": "测试用户",
    "avatarUrl": null,
    "status": 1,
    "createdTime": "2026-03-20T10:00:00",
    "updatedTime": "2026-03-20T10:00:00",
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }
}

失败响应 - 用户名已存在：
{
  "code": 500,
  "message": "用户名已存在",
  "data": null
}

失败响应 - 邮箱已被注册：
{
  "code": 500,
  "message": "邮箱已被注册",
  "data": null
}

失败响应 - 两次密码不一致：
{
  "code": 500,
  "message": "两次输入的密码不一致",
  "data": null
}

失败响应 - 参数校验失败：
{
  "code": 400,
  "message": "username: 用户名长度必须在 3-20 个字符之间",
  "data": null
}
```

##### 错误码说明

| 错误码 | 说明                    |
| ------ | ----------------------- |
| 200    | 成功                    |
| 400    | 参数校验失败            |
| 401    | 未授权/Token 无效或过期 |
| 500    | 服务器内部错误/业务异常 |

---

##### 注意事项

1. JWT Token 使用：
   - 登录成功后会返回 token
   - 需要登录的接口必须在请求头中携带 token
   - 格式：Authorization: Bearer {token}
   - token 有效期为 24 小时

2. 密码安全：
   - 密码使用 BCrypt 加密存储
   - 前端传输明文密码即可
   - 修改密码时需要提供旧密码验证

3. 数据校验：
   - 所有字段都有相应的长度和格式校验
   - 校验失败会返回 400 状态码和具体错误信息

4. 邮箱唯一性：
   - 注册时检查邮箱是否被注册
   - 修改时可以修改为自己的邮箱（允许重复提交自己的邮箱）
   - 不能修改为其他用户的邮箱

5. 用户名不可修改：
   - 用户名在注册后不可更改
   - 只能修改昵称、邮箱、头像和密码

### 架构：

```text
                    前端应用层
                (Vue/React/移动端)
                         │
                    HTTP/HTTPS
                         ▼
                      网关层
               (CORS、JWT 认证拦截)
                         │
                         ▼
                   Controller 层
          (接收请求、参数校验、响应返回)
                         │
                         ▼
                    Service 层
             (业务逻辑处理、事务管理)
                         │
                         ▼

         (数据访问、MyBatis-Plus 操作)
                         │
                         ▼
                   Database 层
                  (MySQL 数据库)
```

####  分层架构说明

##### Controller 层
- **职责**: 接收 HTTP 请求，参数校验，调用 Service，返回响应
- **规范**: 
  - 使用 `@RestController` 注解
  - 统一返回 `Result<T>` 格式
  - 使用 `@Validated` 进行参数校验
  - 不处理业务逻辑，仅做参数传递和响应封装

##### Service 层
- **职责**: 业务逻辑处理，事务管理，数据校验
- **规范**:
  - 使用 `@Service` 注解
  - 使用 `@Transactional` 管理事务
  - 抛出 `BusinessException` 处理业务异常
  - 复杂业务拆分为多个方法

##### Mapper 层
- **职责**: 数据库操作，SQL 执行
- **规范**:
  - 继承 `BaseMapper<T>`
  - 优先使用 MyBatis-Plus 自带方法
  - 复杂查询自定义 SQL 方法
  - 使用 `@Mapper` 注解

修改信息流程:

2. 验证旧密码（修改密码时）

4. 更新用户信息

##### 3.2 知识库模块

###### 功能描述
- 提供知识库的创建、查询、修改、删除能力，作为文件夹、笔记、标签与双链关系的归属边界
- 支持按用户维度查询知识库列表，并按最近更新时间排序返回
- 支持获取知识库目录树，将多级文件夹、根目录笔记和笔记标签整合为统一树形结构
- 在笔记、文件夹发生变更时同步刷新知识库更新时间，为目录树、检索与图谱模块提供最近变更依据
- 删除采用软删除策略，并级联处理知识库下的文件夹、笔记、标签及相关关联数据

###### 核心类
- `KnowledgeBaseController`: 知识库相关接口
- `KnowledgeBaseService`: 知识库业务逻辑
- `KnowledgeBaseMapper`: 知识库数据访问
- `FolderMapper`: 文件夹数据访问
- `NoteMapper`: 笔记数据访问

###### 数据表
- `t_knowledge_base`: 知识库表
- `t_folder`: 文件夹表
- `t_note`: 笔记表
- `t_note_content`: 笔记正文表

###### 业务流程

创建知识库流程:
1. 解析 JWT Token 获取 userId
2. 去除名称首尾空白并校验当前用户下是否重名
3. 写入知识库记录，初始化有效状态与软删除标记
4. 返回知识库视图对象

更新知识库流程:
1. 解析 JWT Token 获取 userId
2. 查询知识库并校验归属关系与有效状态
3. 校验新名称是否与当前用户其他知识库重名
4. 更新知识库名称与描述
5. 返回更新后的知识库信息

删除知识库流程:
1. 解析 JWT Token 获取 userId
2. 查询知识库并校验归属关系
3. 查询知识库下所有有效文件夹与笔记
4. 将笔记与文件夹标记为已删除，写入 `deletedTime` 与 `deleteToken`
5. 清理笔记的源双链关系、笔记标签关系，并删除该知识库下的标签
6. 将知识库自身标记为软删除状态

获取目录树流程:
1. 校验知识库访问权限
2. 查询知识库下所有有效文件夹和有效笔记
3. 先构建文件夹节点映射，再组装父子文件夹层级
4. 将根目录笔记或文件夹内笔记挂接到对应节点
5. 递归排序后返回树形结构数据

##### 3.3 文件夹模块

###### 功能描述
- 提供文件夹创建、修改、删除能力，并支持多级嵌套目录结构
- 支持通过修改 `parentId` 完成文件夹移动，满足知识库内部重组场景
- 对同一父目录下的文件夹名称做唯一性校验，避免同层级重名
- 删除文件夹时会递归处理其子文件夹及其下笔记，并同步刷新失效双链状态

###### 核心类
- `KnowledgeBaseController`: 文件夹相关接口（与知识库共用）
- `KnowledgeBaseService`: 文件夹业务逻辑（与知识库共用）
- `FolderMapper`: 文件夹数据访问
- `NoteMapper`: 文件夹删除时关联笔记数据访问
- `NoteLinkMapper`: 文件夹删除后失效双链刷新
- `NoteTagMapper`: 文件夹删除后笔记标签解绑

###### 数据表
- `t_folder`: 文件夹表
- `t_note`: 笔记表

###### 业务流程

创建文件夹流程:
1. 解析 JWT Token 获取 userId
2. 校验知识库是否存在且归属于当前用户
3. 校验父文件夹是否属于同一知识库
4. 检查当前父目录下是否存在同名文件夹
5. 创建文件夹记录并刷新知识库更新时间
6. 返回文件夹树节点信息

移动文件夹流程:
1. 解析 JWT Token 获取 userId
2. 查询原文件夹并校验所属知识库权限
3. 检查目标父文件夹是否存在且属于同一知识库
4. 校验不能移动到自身或自己的子孙目录下
5. 校验目标父目录下名称是否冲突
6. 更新 `parentId` 与名称，返回最新节点信息

删除文件夹流程:
1. 解析 JWT Token 获取 userId
2. 查询目标文件夹并校验访问权限
3. 收集当前文件夹及全部子文件夹 ID
4. 将目录下所有笔记标记为软删除，并清理其源双链与标签绑定关系
5. 将当前文件夹及其子文件夹统一标记为软删除
6. 刷新知识库内相关双链失效状态，并更新知识库时间戳

##### 3.4 笔记模块

###### 功能描述
- 提供笔记的创建、详情查询、修改、删除能力，正文内容独立存储于 `t_note_content`
- 支持 Markdown 内容编辑，笔记可以直接创建在知识库根目录，也可以归属于指定文件夹
- 支持通过模板初始化新笔记内容，并在详情接口中返回标签、附件、出链和入链信息
- 支持标题自动保存与正文自动保存，减少编辑过程中的内容丢失风险
- 支持手动创建、查看、恢复和删除历史版本，满足重要内容回溯需求
- 删除采用软删除策略，并在标题变更、内容变更、删除后同步刷新双链关系与失效状态

###### 业务流程

创建笔记流程:
1. 解析 JWT Token 获取 userId
2. 校验知识库是否存在且归属于当前用户
3. 校验目标文件夹是否存在并属于同一知识库
4. 检查当前知识库下是否已存在同名笔记
5. 若传入 `templateId` 则读取模板内容，否则使用请求中的 `markdownContent`
6. 创建笔记元信息与正文记录
7. 解析正文中的 `[[...]]` 双链并写入关联关系
8. 刷新知识库更新时间并返回完整笔记详情

更新笔记流程:
1. 解析 JWT Token 获取 userId
2. 查询笔记并校验归属关系、有效状态
3. 校验目标文件夹合法性与标题唯一性
4. 更新笔记标题、所属文件夹和正文内容
5. 重新解析双链，刷新失效链接状态
6. 更新知识库时间并返回最新详情

自动保存流程:
1. 标题自动保存接口校验标题非空且不与同知识库下其他笔记重名
2. 正文自动保存接口会先比对内容是否变化，未变化时直接返回
3. 标题或正文变更后刷新双链关系与笔记/知识库更新时间

历史版本流程:
1. 手动创建快照时，先读取当前笔记标题和正文
2. 若与最新历史版本内容完全一致，则直接复用最新版本，避免重复快照
3. 若内容发生变化，则递增 `versionNo` 并写入 `t_note_history`
4. 恢复历史版本时复用笔记更新流程，将历史标题和正文写回当前笔记
5. 删除历史版本时仅删除目标历史记录，不影响当前笔记正文

#### 表名：`t_note_tag` 笔记标签关联表

**数据库功能：** 存放笔记与标签之间的关联关系，实现标签分类管理。同一篇笔记不能重复关联同一个标签。

加唯一性约束：`(note_id, tag_id)`

| 字段名       | 类型     | 长度 | 键              | 描述     |
| ------------ | -------- | ---- | --------------- | -------- |
| id           | bigint   | 20   | 主键            | 主键id   |
| note_id      | bigint   | 20   | 外键 / 组合唯一 | 笔记id   |
| tag_id       | bigint   | 20   | 外键 / 组合唯一 | 标签id   |
| created_time | datetime | -    |                 | 创建时间 |

```sql
-- =========================
-- 11. 笔记标签关联表 t_note_tag
-- =========================
DROP TABLE IF EXISTS `t_note_tag`;
CREATE TABLE `t_note_tag` (
  `id` BIGINT NOT NULL COMMENT '主键id',
  `note_id` BIGINT NOT NULL COMMENT '笔记id',
  `tag_id` BIGINT NOT NULL COMMENT '标签id',
  `created_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_t_note_tag_note_tag` (`note_id`, `tag_id`),
  KEY `idx_t_note_tag_note_id` (`note_id`),
  KEY `idx_t_note_tag_tag_id` (`tag_id`),
  CONSTRAINT `fk_t_note_tag_note_id`
    FOREIGN KEY (`note_id`) REFERENCES `t_note` (`id`)
    ON DELETE CASCADE
    ON UPDATE RESTRICT,
  CONSTRAINT `fk_t_note_tag_tag_id`
    FOREIGN KEY (`tag_id`) REFERENCES `t_tag` (`id`)
    ON DELETE CASCADE
    ON UPDATE RESTRICT
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='笔记标签关联表';
```

#### 表名：`t_note_template` 模板表

```sql
-- =========================
-- 12. 模板表 t_note_template

DROP TABLE IF EXISTS `t_note_template`;
```
